# Інструкція: Використання мульти-сесій для обходу FloodWait

## Що було додано?

Система тепер підтримує **автоматичну ротацію між декількома Telegram акаунтами** для обходу FloodWait обмежень.

### Основні можливості:
- ✅ Автоматичне виявлення всіх .session файлів в папці `accs/`
- ✅ Автоматичне переключення на наступну сесію при FloodWait
- ✅ Tracking заблокованих сесій в SQLite (стан зберігається між запусками)
- ✅ Рекурсивний пошук доступних сесій
- ✅ Детальні логи та статус-репорти

## Як використовувати?

### 1. Підготовка сесій

Додайте декілька `.session` файлів в папку `accs/`:

```
TelegramAnalysis/
├── accs/
│   ├── account1.session
│   ├── account2.session
│   ├── account3.session
│   └── ... (додайте стільки скільки потрібно)
```

**Важливо:**
- Всі сесії повинні бути **авторизовані** через Telegram
- Всі сесії повинні використовувати **ті самі API_ID та API_HASH** (з .env файлу)

### 2. Запуск системи

Запустіть як зазвичай:

```bash
# Через START_ALL.bat
START_ALL.bat

# Або вручну
venv\Scripts\python.exe analyze_chats_playwright.py --auto
```

### 3. Що відбувається при FloodWait?

Коли одна сесія отримує FloodWait:

1. **Автоматичне блокування:** Сесія позначається як заблокована на N секунд
2. **Автоматичне переключення:** Система миттєво переключається на наступну доступну сесію
3. **Продовження роботи:** Обробка чатів продовжується без затримки
4. **Persistent state:** Стан блокувань зберігається в `state/chats.db`

## Структура нових файлів

### 1. session_manager.py
Новий модуль для управління сесіями:

**Основний клас:** `SessionManager`
- `__init__(db_path, accs_dir)` - ініціалізація
- `get_next_available_session(api_id, api_hash)` - отримати наступну доступну сесію
- `mark_session_blocked(session_name, wait_seconds, error_msg)` - позначити сесію як заблоковану
- `get_available_count()` - скільки сесій доступно зараз
- `get_total_count()` - загальна кількість сесій
- `get_status_report()` - детальний статус всіх сесій

**База даних:** `state/chats.db` (нова таблиця `session_blocks`)

### 2. Модифікації analyze_chats_playwright.py

**Додано:**
- Імпорт `SessionManager, SessionInfo`
- Функція `reconnect_telegram_session()` - переключення сесій
- Try-except FloodWaitError в основному циклі з автоматичним переключенням

**Змінено:**
- КРОК 3: Тепер використовує `SessionManager` замість `get_first_session()`
- Основний цикл обробки: Автоматично ловить FloodWaitError і переключає сесії

## FAQ

### Q: Скільки сесій потрібно?
**A:** Залежить від навантаження. Рекомендації:
- **1-2 сесії:** для невеликих обсягів (< 500 чатів/день)
- **3-5 сесій:** для середніх обсягів (500-2000 чатів/день)
- **5+ сесій:** для великих обсягів (> 2000 чатів/день)

### Q: Що якщо всі сесії заблоковані?
**A:** Система зупиниться і покаже статус:
```
[X] ВСІ TELEGRAM СЕСІЇ ЗАБЛОКОВАНІ FloodWait!

[i] Статус сесій (0/3 доступно):
   [X] account1.session - заблоковано ще на 545хв
   [X] account2.session - заблоковано ще на 320хв
   [X] account3.session - заблоковано ще на 180хв

💡 Додайте більше .session файлів в accs/ або зачекайте розблокування
```

Рішення:
- Додати більше .session файлів
- Зачекати розблокування
- Перезапустити пізніше (стан збережеться)

### Q: Чи зберігається стан блокувань між запусками?
**A:** Так! Стан блокувань зберігається в `state/chats.db`. Якщо ви зупините і перезапустите скрипт, система пам'ятатиме які сесії заблоковані.

### Q: Як створити нові .session файли?
**A:**
1. Скопіюйте існуючий скрипт авторизації (наприклад, з ChatsSpider)
2. Або використайте Telethon вручну:
```python
from telethon import TelegramClient

api_id = YOUR_API_ID
api_hash = "YOUR_API_HASH"

client = TelegramClient('accs/new_account.session', api_id, api_hash)
client.start()  # Введіть номер телефону та код
```

### Q: Чи потрібно щось змінювати в .env файлі?
**A:** Ні! Всі сесії використовують API_ID та API_HASH з існуючого .env файлу.

## Технічні деталі

### Таблиця session_blocks в БД:

```sql
CREATE TABLE session_blocks (
    session_name TEXT PRIMARY KEY,
    blocked_until TEXT NOT NULL,
    last_error TEXT
)
```

### Алгоритм вибору сесії:

1. Завантажити всі .session файли з `accs/`
2. Завантажити стан блокувань з DB
3. Перевірити поточний час - розблокувати сесії де час вийшов
4. Вибрати першу доступну сесію (round-robin)
5. При FloodWait - зберегти блокування в DB і перейти до п.4

### Рекурсивна обробка помилок:

`reconnect_telegram_session()` рекурсивно шукає доступну сесію:
- Якщо сесія не авторизована → блокує на 1 годину → пробує наступну
- Якщо помилка підключення → блокує на 10 хвилин → пробує наступну
- Якщо всі заблоковані → повертає None

## Troubleshooting

### Помилка: "Немає .session файлів в accs/"
**Рішення:** Додайте хоча б один .session файл в папку `accs/`

### Помилка: "Сесія X не авторизована"
**Рішення:** Перевторізуйте сесію через Telethon

### Помилка: "Всі сесії заблоковані"
**Рішення:** Додайте більше сесій або зачекайте

### Логи містять крякозябри (кирилиця)
**Причина:** Windows кодування консолі
**Вплив:** Тільки візуальний, функціонал працює
**Рішення:** Не критично, можна ігнорувати або змінити кодування консолі на UTF-8
